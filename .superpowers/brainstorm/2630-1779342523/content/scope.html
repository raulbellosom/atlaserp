Tabla de alcance: donde aplicar la feature

El sistema actual tiene dos tipos de implementacion de tablas. La configuracion de vistas afecta a cual implementacion?

Arquitectura actual de tablas en Atlas ERP
Modulo Tipo Usa AtlasTable Implementacion
custom.fleet (Flota) AME3 Custom SI — AtlasCrudView → AtlasTable Blueprint-driven, declarativo
Futuros modulos community AME3 Custom SI — misma arquitectura Blueprint-driven, declarativo
atlas.contacts (Contactos) Core Legacy NO — ContactsTableView custom Componente propio con explorer hook
atlas.files (Archivos) Core Legacy NO — FilesTableView custom Componente propio con explorer hook
atlas.finance (Finanzas) Core Legacy NO — HTML table hardcoded TanStack Query + table HTML puro
atlas.hr (RRHH) Core Legacy NO — tabla propia en HrScreen Componente propio

AtlasTable.jsx tiene 799 lineas. Si la feature va en AtlasTable, los modulos legacy necesitarian ser migrados separadamente para beneficiarse.

Cual es el alcance inicial?

Elegir bien aqui define cuanto trabajo implica la primera iteracion.

A

Solo AtlasTable (AME3)

Implementar en AtlasTable unicamente. Cubre custom.fleet y todos los modulos nuevos de la community. Los modulos legacy (contacts, files, finance, HR) quedan fuera por ahora. Menor alcance, implementacion mas limpia y rapida.

B

AtlasTable + adaptar modulos legacy

Implementar en AtlasTable Y migrar los modulos legacy para usar un hook compartido de configuracion de vistas. Mas trabajo, pero consistencia total en toda la app. Probablemente require 2-3x mas tiempo.

C

Hook compartido desde el inicio

Diseniar un hook useTableViewConfig reutilizable que funcione en AtlasTable Y pueda adoptarse progresivamente en modulos legacy sin migrarlos todos ahora. Los legacy adoptan el hook a su ritmo.
